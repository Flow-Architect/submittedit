# 0002: Status language

- Status: accepted
- Date: 2026-07-14
- Scope: Goal 02 evidence language

## Decision

SubmittedIt exposes exactly seven user-facing evidence states. A state describes the strongest evidence that has passed verification, not the user's intention or the most optimistic interpretation of a website response.

**Pending acceptance is the dominant warning in the normal lifecycle.** It must remain visually and verbally prominent whenever no verified authoritative outcome exists, including after a site confirmation. **Verification failed is an exceptional integrity override.** When applicable, it replaces an otherwise optimistic display because the receipt cannot be trusted.

## Display rules

1. Show **Verification failed** if any applicable decryption, schema, hash, signature, continuity, chain, or authority-signature check fails.
2. Otherwise show **Accepted** or **Rejected** only when the authoritative acknowledgment and its linkage verify.
3. A **Site confirmed** event does not clear **Pending acceptance**. Present the site evidence as a completed event and keep the missing-authority warning dominant.
4. An **Attempted** event does not imply the site processed the request.
5. **Prepared** is local readiness, not an anchored submission event.

Labels are never shortened to “Pending,” “Success,” “Done,” or “Failed.” Do not convey state by color alone. In compact surfaces, preserve the exact label and symbol before secondary description.

## Authority and chain language

Accepted and Rejected name the runtime authority that signed the acknowledgment. The fictional demo authority must be identified as such. Monad may independently timestamp and help verify lifecycle fingerprints, but an onchain record does not replace an authoritative acknowledgment or establish legal timeliness.
